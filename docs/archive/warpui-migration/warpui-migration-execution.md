- **File:** `src/state/wake.rs` (new); re-export from `src/state/mod.rs`.
- **Change:**
  ```rust
  pub trait Wake: Send + Sync { fn wake(&self); }
  impl<F: Fn() + Send + Sync> Wake for F { fn wake(&self) { self() } }
  ```

## 2. PHASE 1 — warpui SHELL IN THE REAL CRATE (real data, not placeholders)

### Coexistence strategy — RECOMMENDATION: **Cargo feature flag + two `[[bin]]` targets in the `crane` package**, NOT a separate crate.
- `Cargo.toml`:
  ```toml
  [features]
  default = ["egui_frontend"]
  egui_frontend = ["dep:eframe", "dep:egui", "dep:egui-phosphor"]
  warpui_frontend = ["dep:warpui"]

  [dependencies]
  eframe = { version = "0.34", features = ["wgpu","persistence"], optional = true }
  egui = { version = "0.34", optional = true }
  egui-phosphor = { version = "0.12", optional = true }
  warpui = { path = "vendor/warp/crates/warpui", optional = true }

  [[bin]]
  name = "crane"        # egui — current default
  path = "src/main.rs"
  required-features = ["egui_frontend"]

  [[bin]]
  name = "crane_warp"   # warpui — new frontend under construction
  path = "src/main_warp.rs"
  required-features = ["warpui_frontend"]
  ```
  Rationale: both binaries link the **same** `src/state/**`, `src/git.rs`, `src/git_log/{data..state}.rs`, `src/jobs/**`, `src/file_watcher.rs`, `src/dir_cache.rs` — the Phase-0-decoupled logic. No code duplication, no submodule juggling, and `make ship` keeps building `crane` (egui) until cutover, when you flip `default` to `["warpui_frontend"]` and rename. A second crate would force the shared logic into a third lib crate — more churn for no gain.

## 3. EDITOR DECISION — FINAL CALL

**Build a rope editor (ropey + custom warpui painter). Do NOT use `vendor/warp/crates/editor` (`warp_editor`).** Staged: keep current text editing now → introduce a Crane-owned `CraneTextBuffer` trait (Phase 0/2) → swap in a `ropey`-backed painter later.

Two disqualifying blockers from the editor-crate evaluation, both verified in the map:
1. **Licensing:** `warp_editor` inherits `license.workspace = true` from a workspace whose package license is `AGPL-3.0-only` (`vendor/warp/Cargo.toml:27`, `vendor/warp/LICENSE-AGPL`). Linking it makes the shipped Crane binary a derivative work obligated to ship under AGPL — disqualifying for a closed product.
2. **Rendering/model coupling:** the editor renders **only** through `warpui_core` (GPUI/Zed-style retained framework) — `RichTextElement<V: EditorView> impl Element for ... ` (`render/element/mod.rs:974`) paints via warpui_core `LayoutContext/PaintContext`, not via the warpui surface Crane will host. Every mutating method (`insert/backspace/undo/move_*/select_*`, even `search.rs:88`) takes `&mut ModelContext<Self::T>`, so editing/selection/undo/search cannot be driven outside Warp's reactive app loop. Note this is `warpui_core`, distinct from the `warpui` framework the spike already hosts — adopting it means embedding a *second* GPU UI framework.

The only framework-agnostic piece (`content::buffer::Buffer`, `sum_tree` rope + undo + anchors, `buffer.rs:514-600`) is a **design reference, not a drop-in** (still AGPL source, still pulls `sum_tree`/`string_offset`/`line-ending`). Cost to match it on `ropey` (MIT) ≈ 1-2k lines — cheaper than an AGPL obligation plus a second renderer.

**Action:** `ropey` (MIT) in `Cargo.toml`; `CraneTextBuffer` trait abstracting `load/text/apply_edit/undo/redo/selection/line↔offset/dirty + monotonic content-version` (for LSP didChange/didSave sequencing, mirroring Warp's `ContentVersion`). syntect stays as the span source (`src/views/highlight.rs` emits `(byte_range, color)` spans → convert at paint edge). Find/replace stays Crane-side (`src/views/file_find.rs` + regex over the rope) — never `warp_editor::search`.

## 5. RISKS + SINGLE MOST LIKELY FAILURE

| Risk | Severity | Mitigation |
|---|---|---|
| **DnD feel divergence** in the 3 feel-critical files (`pane_view`/`projects`/`explorer`) — warpui has no `egui::DragAndDrop::payload`/`take_payload` equivalent; payload-based drag + pointer-pos drop-preview must be rebuilt | High | Lift drop-intent resolution (`DockEdge`, `TreeDrag/DropZone`, `FsDropOp`) into **pure functions** during Phase 0 so warpui only feeds pointer-pos+rects and paints the preview. Test the pure resolvers with unit tests before wiring render. |
| **Two GPU frameworks coexisting** (egui/wgpu + warpui) if RenderHandles sharing is wrong | Med | Keep them as separate `[[bin]]`s — never run both in one process. Don't attempt surface-sharing; warpui owns its own window. |
| **`crane_term` sync-replay regression** when PTY reader's `ctx.request_repaint()` → `wake.wake()` | Med | The 38 `crane_term` unit tests are render-agnostic; run `make test` after step 0.6. Wake fires identically. |
| **LSP version sequencing** when editor swaps to ropey | Med | `CraneTextBuffer` exposes a monotonic content-version; keep `file_save.rs` didSave logic untouched. |

**THE single most likely thing to go wrong:** **Step 0.3 (moving `egui::TextureHandle` out of the model) silently breaks image file/diff rendering**, because the field is `#[allow(dead_code)]` in the model and only *written* by the view — so removing it compiles clean and you won't notice until you open a `.png` tab or an image diff and get a blank pane (no compile error, no panic).

**Mitigation:** before Step 0.3, open one image **file tab** and one image **diff** in the running egui app and screenshot them. After the edit, re-open the exact same two and diff the screenshots. Add an assertion in the new view-side cache path (`debug_assert!` on cache hit after first insert) and keep the cache keyed by the same identity the model used (`path` for `FileTab`, `(left_path,right_path)` for `DiffTabData`) so eviction semantics match. Do this step **alone in its own commit** (0.3) so a bisect lands exactly on it if regression appears.
