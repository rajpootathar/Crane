# Crane warpui — faithful port checklist (from old code)

## Process

This is a **faithful port**, not a rewrite. Every feature below must be ported from the old implementation — copied behavior-for-behavior from the named old source — **not approximated, not "close enough," not reinvented from the warpui side.** When old Crane leans on egui 0.34 primitives (the `TextEdit` buffer/cursor/undo/clipboard substrate, `DragAndDrop` payloads, per-widget `context_menu`, `animate_bool_with_time`), warpui has no equivalent and the substrate must be **built**, not stubbed. Concrete rules:

- **Match exact constants, colors, glyphs, sizes, and RGBs** from the old code (row heights, padding, palette RGBs, icon names). Where the new port already invented different values (e.g. `project_tint()` stand-in palette, `"Terminal N"` naming, hardcoded "Menlo"), replace them with the old values — do not keep the divergence unless it is explicitly chosen and documented.
- **Match exact semantics**, including asymmetries (e.g. the × close path confirms via modal, but Cmd+Shift+W closes without confirm — both must be preserved).
- **Do not invent features the old code lacks.** Notably: old Crane has **no tab drag-to-reorder for Workspace Tabs** — the OLD tab row uses `Sense::click_and_drag()` but the drag is never wired to swap `pane.tabs`; there is only click-to-activate and middle-click/× close. Do **not** add reorder to the Workspace Tab strip in the name of "parity."
- **Cite the old source** at each step so the porter can diff against the original.
- **Per-tab focus is the load-bearing invariant** that prevents keystroke misrouting; treat it as a correctness requirement, not a nicety (see Pane Focus & Keyboard).

File-path note carried from the analyses: there is **no `src/ui/left.rs`** and **no `src/ui/right.rs`**. Left panel = `src/ui/projects.rs`; right-panel git = `src/ui/explorer.rs`. The Workspace-level Tab strip does not exist as a horizontal strip in OLD — Tabs render as vertical rows inside the Left Panel tree (`projects.rs`). The new warpui port reorganizes some of these into a mid-pane horizontal `tab_strip` in `shell.rs` — a deliberate divergence noted per area.

## 1. File editor

**Old:** `src/views/file_view.rs` (2074 lines) + `file_find.rs`, `file_save.rs`, `file_status.rs`, `file_util.rs`; state in `src/state/layout.rs` (`FileTab`, `TabKind`, `FilesPane`).
**New:** `crates/warp_term_spike/src/file_pane.rs` (169 lines).

### Old → New gap

The new `file_pane.rs` is a **read-only line viewer** (its own doc comment admits "v1 is read-only"). It models `OpenFile { name, lines: Vec<String> }` — **truncated to 2000 lines** (silent data loss), no `path`, no `content` string, no dirty baseline, no flags. Font is hardcoded **"Menlo"** (not the bundled JetBrains Mono, not `font_size`-configurable). `open()` matches on **basename, not path** (same-name files in different dirs collide). The tab strip has no close, no middle-click, no dirty/diff/RO icons, no hover, no scroll. The body is a flat `Flex::column` of `Text` per line — no gutter, no syntect colors, no line numbers, no real scroll, no status strip, no path row.

It is missing essentially the **entire** `file_view.rs` surface: no editing at all, no save/Cmd+S, no dirty tracking, no find/replace, no goto-line, no undo/redo, no cut/copy/paste, no syntect, no gutter/git-markers/current-line band, no read-only-vs-editable distinction, no external drop, no external-change detection, no markdown preview, no image/PDF, no diff tabs, no context menu, no status strip, no close/reorder, no font sizing, no scroll-to-line.

**The single hardest item is the editable text widget + key-intercept layer** (step 5): old Crane leans entirely on egui 0.34's `TextEdit` for buffer, cursor, selection, undo/redo, and clipboard. warpui has no `TextEdit` equivalent — that whole substrate must be built from scratch, and steps 6–8 depend on it.

## 3. Pane focus & keyboard (correctness — fixes wrong-tab typing)

**Old:** `src/state/layout.rs` (`Layout.focus`), `src/ui/pane_view.rs`, `src/terminal/view.rs`, `src/shortcuts.rs`, `src/state/state.rs`.
**New:** `crates/warp_term_spike/src/{shell.rs, view.rs, layout.rs, input.rs}`.

### Old → New gap

OLD focus is **scoped per Tab**: each Tab owns its `Layout` with `pub focus: Option<PaneId>` (`layout.rs:593`). Only `active_layout()` is ever rendered (`main.rs:698-715`), and `render_terminal` bails `if !has_focus { return None; }` **before any key handling** (`view.rs:1347-1349`) — so a non-focused/non-active pane's PTY is structurally never written to. OLD also swallows raw keys whenever any egui widget owns focus (`other_widget_focused`, `view.rs:1354`) and skips all routing when a modal disables the ui (`input_enabled`, `view.rs:1362`).

NEW moved focus + routing up into the shell with a **single global `focused: Option<PaneId>`** (`shell.rs:42`), while `panes: HashMap<PaneId, PaneContent>` holds panes from **every** tab. `SendKeys` (`shell.rs:1541-1547`) routes to `self.focused.and_then(terminal_at)` with **no check that `focused` is a leaf of the active tab's layout** — `terminal_at` looks up the flat all-tabs map, so a stale `focused` sends keystrokes to a terminal in a non-visible tab. The mouse-down `FocusPane` dispatch (`shell.rs:1371`) only corrects panes whose rects are currently painted (active tab) and cannot fix a stale pointer until you click a visible pane. Also missing: `other_widget_focused`/modal guard (only `commit_focused` is special-cased), `focus_next`/`focus_prev` (Cmd+[ / Cmd+]), and the inactive-pane dim overlay `rgba(0,0,0,45)`.

### Port steps (ordered)

1. **Tab-scope focus.** Move `focused` out of the single shell field into per-tab storage — a `HashMap<(usize,usize,usize), PaneId>` keyed like `layouts`, or a `focus` field on a per-tab struct. Mirror OLD `Layout.focus` (`layout.rs:593`). This is the structural fix that makes misrouting impossible.
2. **[HARDEST] Guard `SendKeys` / `PasteFocused` / `ClearFocused` on the active tab.** Before routing, resolve focus *through* `self.active_tab` → its layout → verify the target id is a leaf of that layout; if not, fall back to `node.first_leaf()`. This is the actual bug, on the routing hot path, and must stay consistent with the focus-sync at `shell.rs:1646-1649` (which should also key off per-tab focus, not the global field).
3. **On every `active_tab` change, set focus to that tab's stored focus (or `first_leaf`).** Audit all writers of `self.active_tab` (`Select` 1504, `NewTab` 1586, `CloseTab` 1607) so none leaves a stale global. OLD reference: `App::set_active` + `Layout.focus` always a leaf of that tab's tree.
4. **Modal / other-widget guard.** Port `view.rs:1354,1362`: when a non-terminal widget owns keyboard focus (today only `commit_focused`; extend to file-editor, find bar, branch picker, any TextEdit) or a modal is open, swallow raw `SendKeys`. Generalize `commit_focused` into a "some UI widget owns keys" predicate.
5. **Wire `focus_next` / `focus_prev` + Cmd+[ / Cmd+].** Add a `Node` helper collecting leaves in tree order (`layout.rs:143` `leaves(&mut Vec)` already does this) cycling modulo length, exactly like OLD `focus_next`/`focus_prev` (`layout.rs:772-797`). Add `FocusPrev`/`FocusNext` actions and map keys in the shell `on_keydown` Cmd block (`shell.rs:1386`).
6. **Inactive-pane dim overlay.** Add `rgba(0,0,0,45)`, 4px rounding, over non-focused panes (OLD `pane_view.rs:494-500`), gated `!focused && drop_edge.is_none()`. Today only header color changes; the body value-shift is missing.
7. **Verify focus-on-press.** The shell `on_left_mouse_down` → `FocusPane` (`shell.rs:1371`) already matches OLD `pressed_inside` (press, not release — so a drag-to-select started in a non-focused pane still moves focus). Confirm it fires before terminal handlers and, once focus is tab-scoped (step 1), updates *that tab's* focus entry.

**Old:** `src/ui/projects.rs` (entry `render` → `render_tree`), row primitive `src/ui/util.rs` (`draw_row`/`draw_trailing`), state in `src/state/state.rs`, call site `src/main.rs:519`. (There is **no `src/ui/left.rs`**.)

### Old → New gap

New `left_sidebar`/`nav_row` reads a **read-only** `ProjectNode` from `~/.crane/session.json` (a serde_json walk) with **no live App state, no git_status, no tints, no groups, no `missing`/`is_loose`/`display_name`** — `WorktreeNode` has only name/path/tabs. It renders a three-level tree with selection-highlight + expand/collapse, but: no "Add Project…" footer, no `+worktree`/`+tab`/remove (×) trailing buttons, no middle-click close, no inline rename, no F2/Cmd+R, no double-click, no right-click context menus anywhere, no user-set tints (the `project_tint()` palette is a fixed index with **different RGBs** than OLD `PROJECT_TINT_PALETTE`, and worktree/tab have no tint), no folder groups/group headers, no drag-reorder, no git-status badges, no attention pulse, no `is_loose` flattening, no `display_name` `alias (name)`, no active-row accent bar. Selection (`selected: (pi,wi,ti)`) is a single tuple, **not** `app.active` semantics. Tab management actually lives in the mid-pane `tab_strip`/`worktree_tabs`, not the tree (the tree's tab rows are read-only labels).

## GLOBAL ordered to-do (most impactful first)

Execute top-down. Correctness and load-bearing substrate first, then the breadth of each panel, then the hardest discretionary work last.

1. **Tab-scope focus + guard `SendKeys`/`PasteFocused`/`ClearFocused` on the active tab.** (§3 steps 1-3.) This is an active **correctness bug** — keystrokes currently land in invisible tabs. Make focus per-tab and verify the target is a leaf of the active tab's layout. Highest priority of everything here.
2. **Build the editable text widget + key-intercept layer.** (§1 step 5.) The biggest single substrate gap. warpui has no `TextEdit`; the buffer/cursor/selection/undo/clipboard layer and the full editing key-intercept block are the prerequisite for save, find/replace, and goto-line. Start it early because it is long-lead.

Foundational data + threading layer the entire right panel and the left-panel git badges depend on; also fixes the spaced-path `git add` breakage and the silent error-swallowing.
