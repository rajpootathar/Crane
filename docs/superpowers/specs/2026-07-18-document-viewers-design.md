# Document viewers: images, PDF, and markdown

**Date**: 2026-07-18
**Status**: Approved, pending implementation plan
**Supersedes / builds on**: `docs/specs/2026-05-03-pdf-viewer-design.md`

## Summary

Restore and extend Crane's non-text document viewing in the warpui front-end:

1. **Image viewer** — a new Pane content type for raster + vector images.
2. **PDF viewer** — port of old Crane's pdfium-backed viewer.
3. **Diff view image block** — replace the "preview pending" placeholder.
4. **Markdown fixes** — two correctness bugs (silent content loss, no wrapping),
   three port regressions, and the missing rendering features.

These ship as one unit because they share a single new primitive: a warpui image
element built on `warpui_core`'s `ImageCache`. Markdown inline images and PDF page
bitmaps both consume it, so it cannot sensibly be split out.

## Why

The egui → warpui migration silently dropped capability that old Crane had:

| Capability | Old Crane | warpui today |
|---|---|---|
| View an image file | yes (`views/file_view.rs:829`) | **no** |
| View a PDF | yes (`views/pdf_view.rs`, `d578f1a`) | **no** |
| Image diffs | yes (`render_image_block`) | placeholder text |
| Edit a `.md` file | yes (`preview_mode` toggle) | **impossible** |
| Markdown lists nest correctly | yes (`Tag::List` handled) | **no** |

Two further defects are net-new and are *not* regressions — old Crane had them too,
but they are severe enough to fix now (see "Markdown correctness bugs").

Notably, `.md` files currently cannot be edited at all: `shell.rs:10799` routes every
markdown file to a read-only view and returns before reaching the editor path. You
cannot edit your own README in Crane.

## Scope

**In scope**

- Image viewer: `png jpg jpeg gif bmp webp ico svg`, animated GIF, fit-to-window + zoom
- PDF viewer: page nav, zoom presets, drag text-selection, Cmd+C, Open Externally, error panel
- Diff view: side-by-side image comparison
- Markdown: table rendering, paragraph-run reset, soft-wrapping composite paragraphs,
  `Tag::List` restoration, links, inline images, strikethrough, task lists, edit/preview toggle

**Out of scope** (inherited unchanged from the 2026-05-03 PDF spec)

In-document find (Cmd+F), cross-page PDF selection, thumbnails, outline/TOC,
annotations, form filling, print. Also out: markdown HTML passthrough, footnotes,
and a markdown WYSIWYG editor.

## Architecture

Follow the `WarpMarkdownView` precedent exactly. It is an established, working path
and neither new viewer needs to deviate from it.

Two new self-contained warpui `View`s implementing `Entity` + `TypedActionView`:

- `src/warpui/image_view.rs` → `WarpImageView`
- `src/warpui/pdf_view.rs` → `WarpPdfView`

Wired into `shell.rs` at the four sites markdown already uses:

| Concern | Existing site | Change |
|---|---|---|
| Pane content enum | `:909` | add `Image(ViewHandle<_>)`, `Pdf(ViewHandle<_>)` |
| Per-path view cache | `:295` | add `image_views`, `pdf_views: HashMap<PathBuf, ViewHandle<_>>` |
| Extension routing | `:10794` (`is_md`) | add image + pdf branches as peers |
| Render | `:10236` | `ChildView::new(h).finish()` |

No new pane plumbing, splitting, or focus logic is required.

### The image element

**warpui already ships a complete `Image` element** — `warpui_core/src/elements/gui/image.rs`.
Crane has simply never used it. The stale comments at `diff_view.rs:374` and
`welcome_view.rs:6` claiming "warpui has no image element" are wrong and should be deleted.

Build against the element, not the lower-level cache:

```rust
Image::new(source: AssetSource, cache_option: CacheOption)
    .contain()                                   // FitType::Contain — fit-to-window
    .enable_animation_with_start_time(Instant)   // animated GIF playback
    .on_load_failure(err_element)                // decode failure fallback
    .before_load(placeholder)                    // optional
    .on_load_timeout(dur, element)               // optional
```

Also available: `.cover()`, `.stretch()`, `.with_corner_radius()`, `.with_opacity()`,
`.first_frame_preview()`, `.top_aligned()`, `.right_aligned()`.

There is therefore **no need to call `ImageCache` or `scene.draw_image` directly**, and
no need to hand-roll decode, texture upload, aspect fitting, or GIF frame stepping. The
element covers all of it. The lower-level API (`image_cache.rs:863`, `scene.rs:605`)
remains relevant only for the PDF page path below, which supplies generated bytes.

**Constraint that shapes the PDF design**: `StaticImage`'s `img` field is private and
is only constructed inside `warpui_core`. External crates cannot build one from raw
bytes. Assets must therefore enter through `AssetSource` (`assets/asset_cache.rs:106`):

```rust
Async   { id, fetch }              // arbitrary bytes from a factory  -> PDF pages
Bundled { path: &'static str }
LocalFile { path, fingerprint }    // -> image files, markdown inline images
```

`AssetSource::Async` is the sanctioned way to feed runtime-generated bitmaps into the
cache **without patching the vendored warp submodule**, which we explicitly avoid.

## Component 1 — Image viewer

Routing: extension match beside `is_md`, using the existing `IMAGE_EXTS` from
`diff_view.rs:67` extended with `svg`. Hoist `IMAGE_EXTS` / `is_image_path_str`
(`diff_view.rs:142`) into a shared module so the viewer, the diff view, and markdown
all share one definition rather than three.

Render with the shipped `Image` element:

```rust
Image::new(AssetSource::LocalFile { path, fingerprint }, cache_option)
    .contain()                                  // fit-to-window
    .enable_animation_with_start_time(opened_at) // animated GIF
    .on_load_failure(text_element("Couldn't decode image"))
```

- Fit-to-window comes from `.contain()`; zoom reuses the PDF zoom presets for consistency.
- Animated GIF playback is handled by the element — no manual frame stepping.
- Undecodable file uses `.on_load_failure(...)` with old Crane's `"Couldn't decode image"` copy.

## Component 2 — PDF viewer

A 1:1 port of `src/views/pdf_view.rs` from `d578f1a` (653 lines), re-hosted as a
warpui `View`. Carries over unchanged:

- `ZOOM_PRESETS = [0.50, 0.75, 1.00, 1.25, 1.50, 2.00, 3.00, 4.00]`, `DEFAULT_ZOOM = 1.00`
- `PAGE_GAP = 12.0`, `TEXTURE_KEEP_RADIUS = 5` (evicts page textures >5 pages from viewport)
- `PdfTabState`, `Selection`, `PageText`, `CharInfo`
- `get_pdfium()` / `candidate_pdfium_paths()` / `host_arch()` runtime dylib resolution
- Selection stack: `screen_to_pdf_pts` → `char_hit_test` → `handle_selection` →
  `paint_selection_for_page` → `selection_text` → Cmd+C

**Page rendering**: pdfium renders a page → RGBA → encode PNG in memory →
`AssetSource::Async { id: (path, page_idx, zoom), fetch }` → `ImageCache` →
`scene.draw_image`. Keying the cache id on zoom means a zoom change is a deliberate
cache miss that re-renders at the new scale — the correct behavior.

Dependency: re-add `pdfium-render = "0.9"`, ABI-matched to the pinned `chromium/7763`.

## Component 3 — Diff view image block

Replace the guard at `diff_view.rs:375-378` (`"Image file — no text diff (image
preview pending)"`) with a side-by-side old/new image comparison, ported from old

## Component 4 — Markdown

### 4a. Table handling + run reset (correctness — silent data loss)

**This is the highest-priority item in the spec.** Tables do not merely fail to
render; they corrupt the surrounding block stream.

Mechanism, in `markdown_view.rs`:

- `_ => {}` (`:189`) swallows `Tag::Table` / `TableHead` / `TableRow` / `TableCell`
- `Event::Text` (`:164`) *still fires* for text inside every table cell, pushing it into `runs`
- No cell/row end tag flushes `runs`; `TagEnd::Paragraph` (`:151`) never fires inside a table
- `Event::Start(Tag::Paragraph)` is **never handled**, so `runs` is never reset at paragraph start

Consequence: every table's cell text accumulates in `runs` and is either merged into
the next paragraph or dumped at EOF as one giant `Block::Para` by the defensive flush
at `:193`. A table-heavy document renders as one or two runaway lines with the bulk of
its content destroyed.

Fix: handle the table events into a `Block::Table { headers, rows, alignments }`
rendered as a measured column grid, **and** reset `runs` on `Event::Start(Tag::Paragraph)`
so an unflushed accumulator can never leak across blocks.

### 4b. Soft-wrap composite paragraphs (correctness — unreadable output)

`markdown_view.rs:296-300` documents the current behavior as an accepted v1 tradeoff:
plain prose takes a single soft-wrapping `Text` (`:288`, `:315`), while any paragraph
containing inline code, bold, or italic falls back to a horizontal row with
`.soft_wrap(false)` (`:329, 334, 342, 348, 405`) and is clipped at the right edge.

In practice almost every technical paragraph contains a code span, so most of the
document does not wrap.

**Fix: replace the hand-rolled `Flex::row`-of-`Text` with `FormattedTextElement`**
(`warpui_core/src/elements/gui/formatted_text_element.rs`). The row cannot wrap by
construction, which is the root cause. `FormattedTextElement` is purpose-built for
multi-style body text and provides, already:

| Capability | Field / API |
|---|---|
| Wrapping mixed-style text | `disable_text_wrapping` — defaults to `false` |
| Hyperlinks | `HyperlinkUrl`, `HyperlinkPosition`, `hyperlink_font_color` |
| Inline code chips | `inline_code_font_color`, `inline_code_bg_color` |
| Text selection | `text_selection_color`, `FormattedTextSelectionLocation` |
| Heading scaling | `HeadingFontSizeMultipliers::get_multiplier(level)` |

Its own docs direct callers here when they "need multiple styles throughout the text
body, or need to make use of hyperlinks". This one migration resolves 4b and most of
4d together, and adds text selection — which the current view lacks entirely.

Do **not** implement a custom wrapping flow container. That would duplicate shipped
functionality.

### 4c. Restore `Tag::List` (port regression)

warpui handles `Tag::Item` but never `Tag::List`; old Crane handled both
(`views/markdown_view.rs:98/102`). Without list-level tracking there is no nesting
depth, so closing a nested list emits a phantom empty bullet, ordered lists lose their
numbering, and nested items lose indentation. Restore list start/end handling with a
depth counter.

### 4d. Remaining rendering features

`Options::all()` is already set (`:76`), so these events are **already being parsed**
and merely discarded — the work is in the block model and renderer, not the parser.

- **Links** — the URL is currently discarded entirely; link text survives only
  incidentally via `Event::Text`. Once 4b migrates to `FormattedTextElement`, links
  are carried by its native `HyperlinkUrl` / `HyperlinkPosition` support rather than
  by a bespoke clickable-text implementation.
- **Inline images** — via the Component 1 `Image` element; relative paths resolve
  against the `.md` file's directory.
- **Strikethrough** — add to the inline run attributes.
- **Task lists** — handle `Event::TaskListMarker(bool)` as checkboxes.

Because 4b already brings in `FormattedTextElement`, links and inline-code styling
largely fall out of that migration. Sequence 4b before 4d so the work is not done twice.

### 4e. Edit/preview toggle (port regression)

Port of old Crane's `preview_mode` (`views/file_view.rs:534-545`), whose toggle used
an `icons::EYE` "Preview" affordance.

- The pane swaps between `PaneContent::Markdown` and `PaneContent::Editor` for the
  same path. Both caches are already keyed by `PathBuf`, so both handles are held and
  the pane content is swapped.
- Toggling **to preview** re-parses via `WarpMarkdownView::from_source(ctx, title, text)`
  — which already exists at `:241` marked *"not yet wired"* — fed from the editor's
  **live buffer**, so unsaved edits render immediately.
- **Default: preview-first**, with a toggle to edit. This is a deliberate deviation
  from old Crane (which was editor-first). Rationale: it preserves current warpui
  behavior rather than regressing it, and the toggle resolves the actual complaint.

**Naming**: old Crane had two distinct concepts both called "preview" —
`preview_mode` (render vs edit) and `tab.preview` (the VS Code-style italic tab
promoted to permanent on edit, `file_view.rs:1899`). Keep these names distinct.

## Error handling

| Condition | Behavior |
|---|---|
| Encrypted / corrupt PDF | Error panel + Open Externally. Never a crash. |
| `libpdfium.dylib` missing | Error panel naming the searched paths. Never a crash. |
| Undecodable image | `"Couldn't decode image"` (old Crane behavior) |
| Markdown parse edge cases | Render what parsed; never silently drop a block |

## Build and distribution

**No build work required.** The pdfium plumbing survived the warpui migration intact
and is still wired in the current `Makefile`:

- `vendor-pdfium` (`:87`) → `scripts/vendor-pdfium.sh`, pinned to `chromium/7763`
- `bundle` (`:90`) copies the host-arch dylib into `Contents/Frameworks`, then
  `install_name_tool -id @rpath/libpdfium.dylib` + `-add_rpath @executable_path/../Frameworks` (`:96-100`)
- `bundle-universal` (`:130`) `lipo`-fuses both arch dylibs into one fat dylib (`:140-147`)

`vendor/pdfium/{arm64,x86_64}/libpdfium.dylib` are present locally and gitignored.

A macOS-native PDFKit backend was considered and **rejected**: it would discard this
working, cross-platform-capable infrastructure and lock PDF support to macOS for no gain.

## Testing

**Unit**
- Extension routing for each viewer; `is_image_path_str` already covered (`diff_view.rs:1518`)
- `step_zoom` preset stepping; `char_hit_test` coordinate mapping
- Markdown: a table document round-trips to a `Block::Table` **and leaks no cell text
  into adjacent paragraphs** — this is the regression test for 4a
- Markdown: nested list closes without emitting a phantom empty bullet (4c)
- Markdown: a paragraph containing inline code wraps (4b)

**Diagnostic**
- Restore `examples/pdf_smoke.rs` as the standalone pdfium binding check

**Manual — required**
- Verify PDF inside a real DMG bundle. Dylib resolution differs between `cargo run`
  and a bundled `.app`, and per the browser-pane stale-rect bug, bundle-only failures
  are an established pattern in this codebase. A passing `cargo run` is not evidence.

## Risks and mitigations

| Risk | Mitigation |
|---|---|
| PNG encode→decode round-trip per PDF page per zoom level | Bounded by `TEXTURE_KEEP_RADIUS`. If measurement shows it is too slow, upstream a `StaticImage::from_rgba` into vendored warp — but only if measured, not preemptively. |
| Bundled-only dylib resolution failure | Mandatory DMG verification above. |
| Notarization of the bundled dylib | Confirm `signed-dmg` signs `Contents/Frameworks`, not just the app binary. |
| Composite-paragraph wrapping is a non-trivial layout change | Land 4a and 4b behind their own tests first; they are correctness fixes and are independently valuable. |
| Scope size | Ordered by severity: 4a and 4b are data-loss/readability bugs and should land before any new feature work. |

## Implementation order

1. **4a** table handling + run reset — stops silent content loss
2. **4b** composite paragraph wrapping — makes documents readable
3. **4c** `Tag::List` restoration
4. **Component 1** image element (unblocks 3, 4d)
5. **Component 3** diff image block
6. **4d** markdown links, inline images, strikethrough, task lists
7. **4e** markdown edit/preview toggle
8. **Component 2** PDF viewer

Steps 1–3 are correctness fixes with no dependency on the image element and deliver
value immediately. The PDF viewer is last: it is the largest single piece and the only
one carrying native-dependency risk.
